## 2. `/trust` Page Structure

### Page Title

Trust is a product feature

### Above the Fold

AIResearch is built around a simple rule: every research workflow must
resolve the entity, fetch current data, and degrade visibly when data is
missing.

Primary proof points:

- Entity gates before analysis
- Deterministic technical indicators
- GAAP-first financial data
- ETF-backed Industry Mode
- No trade execution
- No hidden "best guess" substitutions

### Section: The Failure We Designed Around

Summarize the Liquid Silicone Rubber incident and link to the full report:
`/blog/liquid-silicone-rubber-incident-report`.

Pull quote:

> Prompts are suggestions; pipelines are guarantees.

### Section: Data Integrity Controls

1. **Entity Resolution Gate** — every listed-security report starts by
   resolving to exactly one verified entity.
2. **Entity Lock** — every lens receives the same immutable ticker,
   company name, and exchange.
3. **Deterministic Technicals** — RSI, MACD, moving averages, support, and
   resistance are computed from OHLCV data, not generated by the model.
4. **Industry Universe Builder** — theme queries use ETF holdings or
   explicit source-backed search results.
5. **Graceful Degradation** — unresolved or missing data is shown as
   skipped/withheld instead of filled with guesses.

### Section: What AIResearch Will Not Do

- It will not execute, place, modify, or cancel trades.
- It will not write "buy the dip" or "take profit" instructions.
- It will not invent price levels, technical indicators, or financials.
- It will not substitute a similarly named ticker after resolution fails.

### Section: Current Status

- Deep Dive: available
- Snapshot: available
- Industry Mode: available with hosted MCP data
- Morning Brief: rolling out in v0.4
- Per-user dashboard keys: beta/manual issuance

## 3. Blog Post: Liquid Silicone Rubber Incident Report

**Slug:** `/blog/liquid-silicone-rubber-incident-report`

**Title:** The Liquid Silicone Rubber Incident: Why AI Research Needs
Pipelines, Not Just Prompts

**Deck:** A real failure analysis of how a theme query became a ticker
mistake, and how AIResearch changed its architecture afterward.

AI stock research tools fail in a predictable way: they sound confident
before they are grounded. The Liquid Silicone Rubber incident was our
cleanest example.

The user asked for research on "liquid silicone rubber" -- an industry and
material category, not a single listed company. A language model can
understand that phrase semantically, but a stock-research workflow has a
harder job: it must decide whether the input is a ticker, a company, a
theme, a material, or an ambiguous phrase. In the failed version, the
workflow treated the phrase as if it could be coerced into a stock-style
report. Different parts of the analysis began reasoning from different
implicit assumptions. That is the failure mode: one prompt, many hidden
entities.

The fix was not a better instruction like "be careful with tickers." That
helps, but it does not guarantee behavior. Prompts are suggestions;
pipelines are guarantees.

After the incident, AIResearch added an Entity Resolution Gate before any
lens can run. The gate has one job: resolve the user's input to exactly one
verified listed entity, or stop the single-stock workflow. If the input is
a material, industry, or theme, it is routed to Industry Mode instead of
being forced through a stock template. If the input is ambiguous, the
system returns candidates. If it cannot resolve, the analysis is withheld.

That gate is paired with an Entity Lock. Once a ticker is verified, every
research lens receives the same immutable block: ticker, company name,
exchange, quote type, and entity-specific context. Supply Chain cannot
quietly analyze one company while Technicals analyze another. If a lens
lacks data for the locked entity, it must say so.

We also removed another class of hallucination: generated technicals. A
model should not invent RSI, MACD, moving averages, support, or resistance
because those numbers are deterministic. They can be computed from price
history or withheld. In AIResearch, technical indicators come from raw
market data and are injected as a Technical Data Lock. The model can
interpret them, but it cannot create them.

Industry Mode received its own guardrail. For theme queries, AIResearch
uses ETF holdings when available. Theme ETFs are imperfect, but they are
professionally curated baskets with weights. They give the model a bounded
constituent universe. When ETF data is unavailable, the fallback is
source-backed search, not memory.

The result is less magical and more useful. Sometimes the product refuses
to write a report. Sometimes it asks the user to choose from candidates.
Sometimes it says a section is unavailable. Those are not UX failures.
They are trust features.

The lesson from Liquid Silicone Rubber is that finance workflows should
not optimize for the smoothest answer. They should optimize for the safest
path from question to evidence. A fluent report built on the wrong entity
is worse than no report at all.

AIResearch is still an assistant, not an oracle. It does not replace
filings, price feeds, or judgment. But it now treats entity resolution,
data provenance, and graceful refusal as first-class product surfaces.
That is the difference between a prompt that hopes and a pipeline that
checks.
